Problem

Distributed transactions are hard.

Microservices break traditional database transactions. When inventory, payments, shipping, notifications, or provisioning must move together, every service boundary creates another chance for partial failure.

  • Partial failures leave systems in inconsistent states.
  • Retries and compensation logic get duplicated across teams.
  • Workflow coordination becomes hard to test and reason about.
  • Operational visibility is weak when orchestration is spread everywhere.

Solution

Trama coordinates distributed workflows.

Trama is a lightweight saga orchestrator that manages multi-step workflows across services. It gives application teams one place to define how a workflow runs, retries, compensates, persists, and exposes operational signals.

You keep the flexibility of service-based systems without adopting a heavier workflow platform.

RetriesCompensationPersistence MetricsTracingHTTP API
  • Define workflows centrally.
  • Track every execution end to end.
  • Retry or compensate when downstream calls fail.

Real Example

Run a workflow with one API request.

curl -X POST http://localhost:8080/sagas/run \
  -H 'Content-Type: application/json' \
  -d '{
    "definition": {
      "name": "order-fulfillment",
      "version": "v1",
      "failureHandling": {
        "type": "retry",
        "maxAttempts": 3,
        "delayMillis": 500
      },
      "steps": [
        {
          "name": "reserve-inventory",
          "up": {
            "url": "http://inventory/reserve",
            "verb": "POST"
          },
          "down": {
            "url": "http://inventory/release",
            "verb": "POST"
          }
        },
        {
          "name": "charge-payment",
          "up": {
            "url": "http://payment/charge",
            "verb": "POST"
          },
          "down": {
            "url": "http://payment/refund",
            "verb": "POST"
          }
        }
      ]
    },
    "payload": {
      "orderId": "12345",
      "amount": 150.0
    }
  }'

Architecture

Simple flow, clear operational model.

Client→ HTTP API→ Queue→ Workers→ Services

Trama receives a workflow request, persists execution state, pushes work to queue-backed workers, and tracks success, retries, or compensation as the saga runs.

  • Incoming requests become durable workflow executions.
  • Workers claim ready jobs and process steps asynchronously.
  • Retries and compensation are part of the runtime, not hand-coded everywhere.
  • Metrics and traces expose what happened during each run.

Features

Designed for reliable microservices workflows.

Reliable Execution

Define workflows once and run them consistently with queue-backed workers and persistent execution tracking.

Compensation Support

Attach rollback behavior to each step so Trama can unwind workflows when downstream services fail.

Persistence and Visibility

Queue-Native Runtime

Comparison

Positioned between hand-rolled orchestration and heavyweight workflow platforms.

Hand-Rolled

Custom saga logic in every service

  • Lowest upfront tooling cost
  • Highest duplication of retries and compensation
  • Poor cross-workflow visibility
Trama

Focused orchestration with clear operational boundaries

  • Simple setup for microservices workflows
  • Built-in retries, compensation, persistence, and observability
  • Best fit for teams that want orchestration without workflow-platform overhead
Heavy Engines

Broad workflow platforms

  • More features and deeper platform scope
  • Higher operational and adoption complexity
  • Best fit when workflow management becomes a platform in itself
