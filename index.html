Minimal Saga Runtime

Saga orchestration without workflow platform overhead.

Trama is a minimal runtime for coordinating distributed workflows across microservices using HTTP APIs.

It executes saga steps, stores execution state, handles retries and compensation, and exposes execution visibility without requiring a full workflow engine.

Comparison

Saga orchestration without chaos

Distributed workflows become hard to reason about when orchestration logic lives inside individual services.

Hand-rolled saga orchestration

  • Saga orchestration logic spread across multiple services
  • Retry and rollback behavior implemented inconsistently
  • Compensation flows duplicated across teams
  • Execution state buried in logs
  • Hard to reconstruct failures across service boundaries
  • No single place to inspect workflow progress
  • Operational visibility depends on custom tooling

With Trama

  • Dedicated runtime for saga orchestration
  • Consistent retry and compensation policies
  • Durable execution state persisted by the runtime
  • Clear step-by-step execution history
  • API to inspect and debug running workflows
  • Built-in metrics and tracing
  • Explicit failure handling and compensation flows

Trama moves workflow coordination out of services and into a dedicated runtime.

    Architecture

    Runtime model

    Client→ Trama HTTP API→ Execution Store→ Queue→ Workers→ Services→ Termination notice

    Trama separates orchestration from execution. The API receives workflow requests and stores execution state before work is handed to step workers.

    Workers process queued steps while Trama manages retry logic, compensation, and progress tracking.

    Capabilities

    Runtime capabilities

    Trama provides the primitives required to safely coordinate distributed workflows in production.

    Horizontal execution scaling

    Saga steps are processed by workers consuming tasks from queues. Workers can scale horizontally while execution state remains persisted and workers stay stateless.

    Dynamic request templating

    Saga step requests can render values from the execution payload and previous step results directly into HTTP requests.

    "orderId": "{{payload.orderId}}"

    Failure-aware execution

    Retry and compensation behavior are defined in the saga definition. When a step fails, Trama triggers compensating actions for previously completed steps.

    Persistent workflow state

    Execution state is durably stored and can be inspected through the API, including which steps succeeded, failed, or were compensated.

    ✓ reserve-inventory ✓ charge-payment ✗ create-shipment compensation started

    Metrics and tracing

    Trama exposes execution metrics and traces so teams can monitor workflow throughput, failures, retries, and latency across distributed workflows.

    API-first orchestration

    Trama exposes an HTTP API and OpenAPI contract, so workflows can be triggered and inspected from any language or platform without SDK coupling.

    Comparison

    Most teams do not need a workflow platform.

    There are three common approaches to distributed workflows.

    Approach How it works Tradeoff
    Hand-rolled orchestration Workflow logic spread across services with inconsistent retry and rollback behavior. Fast to start, difficult to maintain as workflows grow.
    Trama Focused runtime for distributed workflows executed through a simple API. Centralizes saga coordination without turning workflow management into a platform project.
    Heavy workflow engines Powerful orchestration platforms designed for complex workflow management and platform-level automation. Broader capabilities, but more operational and adoption overhead.
