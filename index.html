Minimal Saga Runtime

Saga orchestration without workflow platform overhead.

Trama is a minimal runtime for coordinating distributed workflows across microservices using HTTP APIs.

It executes saga steps, stores execution state, handles retries and compensation, and exposes execution visibility without requiring a full workflow engine.

Why Trama Exists

Coordination logic should not be spread across every service.

Distributed business workflows often span inventory, payments, shipping, and notifications. When one step fails, the system must retry safely or undo earlier actions.

inventory payments shipping notifications

Without orchestration, retry logic, compensation flows, and state tracking become scattered across services. Trama centralizes this execution logic while keeping service implementations focused on local work.

Problem

Distributed workflows fail in messy ways

In microservice architectures, a single business operation often spans several services. Inventory may succeed while payment times out, and shipping might begin before a failure becomes visible.

These partial failures make distributed workflows difficult to reason about. The Saga pattern addresses this by coordinating local transactions and triggering compensating actions when failures occur.

  • Inconsistent rollback behavior across services
  • Retry logic duplicated across application boundaries
  • Lack of visibility into execution state

Solution

Trama gives sagas a runtime

Trama executes distributed workflows through a persistent runtime. A client submits a workflow request, and Trama stores execution state, schedules step processing, manages retries, and triggers compensation when necessary.

Workers process steps asynchronously while Trama maintains a durable record of the full execution. This separates workflow coordination from service implementation.

Workflow state

  • ✓Inventory reserved
  • ✓Payment charged
  • ⏳Shipment creation in progress

Trama keeps track of which steps already succeeded and which one is currently running, so retries or compensation can happen from a known state.

Example

One request defines the workflow and starts execution.

curl -X POST http://localhost:8080/sagas/run \
  -H 'Content-Type: application/json' \
  -d '{
    "definition": {
      "name": "order-fulfillment",
      "version": "v1",
      "failureHandling": {
        "type": "retry",
        "maxAttempts": 3,
        "delayMillis": 500
      },
      "steps": [
        {
          "name": "reserve-inventory",
          "up": {
            "url": "http://inventory/reserve",
            "verb": "POST"
          },
          "down": {
            "url": "http://inventory/release",
            "verb": "POST"
          }
        },
        {
          "name": "charge-payment",
          "up": {
            "url": "http://payment/charge",
            "verb": "POST"
          },
          "down": {
            "url": "http://payment/refund",
            "verb": "POST"
          }
        }
      ]
    },
    "payload": {
      "orderId": "12345",
      "amount": 150.0
    }
  }'

What happens after this request

1. Persist execution
Trama stores the workflow execution before workers begin processing.
2. Process queued steps
Step workers consume tasks from the queue and execute the workflow asynchronously.
3. Handle failures centrally
Failures trigger retry or compensation depending on configuration.
4. Inspect state later
Execution history remains inspectable through the API and metrics.

This demonstrates that Trama is not just an HTTP wrapper. It provides a persistent orchestration runtime.

Architecture

Runtime model

Client→ Trama HTTP API→ Execution Store→ Queue→ Workers→ Services→ Termination notice

Trama separates orchestration from execution. The API receives workflow requests and stores execution state before work is handed to step workers.

Workers process queued steps while Trama manages retry logic, compensation, and progress tracking.

Comparison

Most teams do not need a workflow platform.

There are three common approaches to distributed workflows.

Approach How it works Tradeoff
Hand-rolled orchestration Workflow logic spread across services with inconsistent retry and rollback behavior. Fast to start, difficult to maintain as workflows grow.
Trama Focused runtime for distributed workflows executed through a simple API. Centralizes saga coordination without turning workflow management into a platform project.
Heavy workflow engines Powerful orchestration platforms designed for complex workflow management and platform-level automation. Broader capabilities, but more operational and adoption overhead.
